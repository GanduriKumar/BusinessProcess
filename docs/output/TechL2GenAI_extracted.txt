Reference Architecture for AI-Enhanced Telecom 
Analytics on AWS
Customer context and objectives
The customer operates telecom hardware that generates large volumes of usage and performance 
data. They want to convert this data into actionable insights using AI/ML, automatically detect 
anomalies and launch remediation workflows, and expose natural-language chatbot interfaces to 
query the data. Two HCLTech products are in scope:
HCLTech eIUM – a usage mediation and management platform that collects, aggregates and 
correlates usage data from telecom infrastructure and exposes the data through RESTful APIs. 
It is distributed and modular, making it well suited for micro-services architectures.
•
HCLTech ART – an automation and reporting tool that provides a workflow designer capable of 
orchestrating non-interactive tasks, a query builder that converts natural-language questions 
into SQL queries, and a dashboard designer for custom reports. ART also includes 
machine-learning-based behaviour and anomaly detection capabilities and can integrate telco 
APIs (CAMARA).
•
The solution must run on AWS and include appropriate services for data ingestion, storage, 
processing, AI/ML modelling, API endpoints, security, role-based access control (RBAC), networking 
and redundancy. The following reference architecture addresses these requirements and follows 
AWS best practices.
Architectural overview
1. On-premises data collection
Telecom hardware & sensors on the customer’s network produce usage records, performance 
metrics and alarms. These devices communicate with HCLTech eIUM, which captures, 
aggregates and correlates usage data from all network layers and publishes the results 
through RESTful APIs. eIUM is designed as a distributed, high-availability application, which 
makes it a good candidate for deployment on a container platform on-premises or at the edge.
•
eIUM sends usage and metric data to AWS through AWS IoT Core and Amazon Kinesis. 
IoT Core allows connected devices or applications to send messages using MQTT/HTTP and 
securely routes messages to AWS services via a rules 
engineemqx.medium.comemqx.medium.com. Kinesis Data Streams/Firehose provides 
durable, serverless streaming of large volumes of real-time datarisingwave.com. Event 
notifications from eIUM can also be published to Amazon EventBridge, a managed event bus 
that routes events from various sources to targets such as Lambda, Step Functions or 
SNSgeeksforgeeks.org.
•
2. Data storage and ETL
Amazon S3 Data Lake stores raw usage and telemetry data. S3 provides durable, scalable 
object storage and integrates with many AWS analytics services.
•
AWS Glue performs serverless ETL; it automatically provisions compute, infers schemas, and 
orchestrates jobsmedium.com. Glue jobs extract data from S3, clean and transform it (e.g., 
flatten nested records), and load it into queryable stores. Glue also maintains a Data Catalog
and offers a visual UI for building pipelinesmedium.com. Jobs can be scheduled or triggered by 
events from Kinesis or EventBridge.
•
Processed data is loaded into both Amazon DynamoDB and Amazon RDS/Aurora. DynamoDB 
is a fully managed, highly available NoSQL database that eliminates manual administration and 
spreads data across servers for fault tolerancemedium.com. It delivers consistent single-digit 
millisecond performance at any scalemedium.com and supports flexible key-value or 
document schemasmedium.com. Multi-region, multi-master replication improves availability 
and disaster recoverymedium.com, and DynamoDB offers encryption at rest, network isolation 
and IAM-based access controlmedium.com.
