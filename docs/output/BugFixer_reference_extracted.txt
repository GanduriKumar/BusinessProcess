Universal Bug-Fixer 2.0 – Current Limitations
Known Issues
Platform & Deployment Constraints
•
Dependent on CLI and IDE tooling; “standalone application” is not supported.
•
“Customer repository access” is mandatory for cloud execution; without agent check-in, cloud runs are not possible
•
Cannot access documents or resources behind customer networks unless explicitly configured via IDE/CLI
Technical Scope Limitations
•
Multi-repo environments (different technologies across repositories under a common root) are not currently supported
•
Third-party library defects cannot be analyzed if the issue originates outside the project codebase
•
PR review comments and post-review iterations are not yet handled automatically
Build & Execution Dependencies
•
Test execution may fail if build instructions are unclear or undocumented in the repository
•
Often relies on developer input to resolve build or environment-specific failures
Bug Complexity & Accuracy Challenges
•
Vague or poorly defined bug descriptions MAY lead to incorrect fix suggestions
•
Highly complex bugs may be mishandled in fully autonomous mode
•
In some cases, the agent may redefine logic instead of reusing existing helper functions, leading to suboptimal fixes
Mitigation & Adoption Strategy
•
To manage these limitations effectively; start with non-critical bugs projects / low complex bugs to build confidence
•
Maintain full transparency through audit trails and change visibility
•
Use SAFE_MODE to control the level of automation
Leverage dedicated support and implementation teams to reduce execution risk

Demo Overview — Autonomous Bug Fixing (Non-Interactive Mode)
Scope & Goal
Automatically detect and execute fixes for assigned Jira bug tickets using Jira & GitLab API integrations and GitHub Copilot, progressing tickets through the workflow 
and producing a merge request (MR) with tests and full traceability—without human prompts. 
1) Agent auto-intakes newly assigned Jira ticket → 2) Retrieves repo context via GitLab APIs → 3) Applies fix using Copilot-assisted changes → 4) Runs tests 
and generates/updates unit tests → 5) Commits code and creates MR → 6) Updates Jira status and adds structured notes/links. 
Mode of Execution
Fully non-interactive (autonomous) run with predefined guardrails; the agent performs code edits, testing, commits, and status transitions while adhering to 
enterprise review gates at MR stage. 
What We’ll Show Live Auto-assignment intake:
Result Variability (What Influences Output)
Defect scope and codebase complexity, existence of similar prior fixes/patterns, and quality of ticket context (logs, repro steps, acceptance criteria) directly impact fix 
accuracy and amount of follow-up review needed. 
Expected Outcome 
A ready-for-review MR containing the bug fix, corresponding unit tests, and complete Jira traceability (status updates, notes, and linked artifacts), produced end-to-end by 
the agent in autonomous mode.
•
Agent detects ticket ownership and moves status to In Progress. 
•
Context gathering: pulls ticket details from Jira and repository/branch data from GitLab via API keys. 
•
Autonomous fix: applies code changes using GitHub Copilot assistance and internal patterns. 
•
Test coverage: generates/updates unit tests, executes them, and records results. 
•
Status automation: transitions Jira ticket In Progress → Unit Testing → In Review with auto-generated notes. MR creation: opens a GitLab MR, links Jira 
issue(s), and attaches test results and change summary.
